**Dificuldade:** L · **Relevância:** crítica

## Abordagem proposta

O objetivo é replicar o que a própria Infuse faz e que a comunidade confirma como padrão de mercado: gerar a miniatura **sob demanda, no momento do scrub**, para qualquer codec/container, sem pré-gerar nem persistir um sprite sheet em disco (ao contrário do modelo Jellyfin/Plex/BIF, que precisa de um passo de indexação prévio — inviável aqui porque o StreamHub reproduz *streams HTTP remotos de debrid*, não arquivos locais indexáveis de antemão). Essa escolha é reforçada pela própria pista que o fork já contém: `ThumbnailController` (`Sources/KSPlayer/MEPlayer/ThumbnailController.swift:24-132`) resolve a decodificação pontual via FFmpeg, mas do jeito errado para scrub ao vivo — ele varre o arquivo inteiro de uma vez (100 seeks sequenciais) em vez de decodificar só o frame perto do tempo que o usuário está arrastando.

Como o fork já tem **dois engines de playback** (`KSAVPlayer` sobre `AVPlayer`/`AVFoundation` e `KSMEPlayer` sobre FFmpeg puro — ver `docs/00-ARQUITETURA.md` e `docs/02-camada-avplayer.md`), a feature precisa de duas implementações técnicas distintas atrás do mesmo contrato público, escolhidas por `KSPlayerLayer` como hoje já acontece com `thumbnailImageAtCurrentTime()`:

1. **Engine `KSAVPlayer` (AVFoundation/HLS)** — praticamente pronto. `KSAVPlayer.swift:573-591` já implementa `createImageGenerator()` + `thumbnailImage(currentTime:handler:)` sobre `AVAssetImageGenerator.generateCGImagesAsynchronously(forTimes:)`, a API padrão da Apple para exatamente este caso de uso (thumbnail em tempo arbitrário, independente da reprodução). Falta: (a) trocar a assinatura de `thumbnailImageAtCurrentTime()` — hoje hardcoded em `playerItem.currentTime()` (`KSAVPlayer.swift:388-397`) — por uma variante que aceita um `TimeInterval` arbitrário; (b) migrar para a API `async` moderna `AVAssetImageGenerator.image(at:)` (disponível a partir de iOS/tvOS 16, mesmo baseline do `KSVideoPlayerView`, ver WWDC22 "Create a more responsive media app"), que já devolve o tempo real gerado e permite cancelamento cooperativo; (c) aumentar `requestedTimeToleranceBefore/After` (hoje `.zero`, linhas 575-576/582-583) para algo como ±0.5-1s durante o scrub — tolerância zero força o gerador a decodificar exatamente aquele frame, o que é lento; para uma prévia de scrub, o frame-chave mais próximo já é suficiente e ordens de magnitude mais rápido (mesma lição do backlog do Jellyfin, ver Referências). Este caminho fica pronto para quando o remux HLS local → AVPlayer (a "joia da coroa" de Dolby Vision) virar o pipeline padrão de reprodução.

2. **Engine `KSMEPlayer` (FFmpeg puro, o caminho hoje usado para MKV/HEVC/HDR10 estático)** — não existe hoje; `thumbnailImageAtCurrentTime()` (`KSMEPlayer.swift:438-440`) só devolve o `pixelBuffer` do frame *já decodificado na posição atual* (usado apenas para uma máscara visual ao trocar de definição, `IOSVideoPlayerView.swift:232-241` — não serve para scrub). A abordagem correta é generalizar o próprio `ThumbnailController`: em vez do laço que gera N thumbnails inteiros de uma vez (`getPeeks`, linhas 94-127), expor um método pontual — algo como `thumbnail(near time: TimeInterval) async throws -> FFThumbnail?` — que reaproveita exatamente a mesma sequência já implementada por iteração do laço (`avcodec_flush_buffers` + `av_seek_frame(AVSEEK_FLAG_BACKWARD)` + `av_read_frame`/`avcodec_send_packet`/`avcodec_receive_frame` + `VideoSwresample.transfer(frame:)` + `cgImage()`), mas para um único ponto por chamada. Isso é o equivalente Swift/FFmpeg do que o backlog do Jellyfin já validou como o caminho rápido: seekar para o keyframe mais próximo (`AVSEEK_FLAG_BACKWARD`) e decodificar 1 frame, sem varrer o GOP inteiro — ~110x mais rápido que decodificar tudo via filtro `fps` (ver Referências). Precisa rodar sobre um `AVFormatContext`/`AVCodecContext` **secundário e persistente** (aberto uma vez quando o player abre, não recriado a cada scrub) para não pagar o custo de handshake HTTP/TLS a cada arrasto — e, criticamente, esse segundo contexto tem que reaproveitar `options.formatContextOptions`/`avOptions` (headers, `user_agent`, `referer`, protocol whitelist) exatamente como `MEPlayerItem.openThread` já faz (`MEPlayerItem.swift:191,202`), porque hoje `ThumbnailController.getPeeks` abre com `avformat_open_input(&formatCtx, urlString, nil, nil)` (`ThumbnailController.swift:49`, opções `nil`) — contra um endpoint de debrid que exige header/token específico isso falha silenciosamente ou recebe 403/401.

3. **Camada de UI (SwiftUI, `KSVideoPlayerView.swift`)** — hoje o scrub bar do tvOS é o `Slider` dentro de `VideoTimeShowView` (`KSVideoPlayerView.swift:561-586`), alimentado por `ControllerTimeModel.currentTime: Int` (`KSVideoPlayer.swift:330-336`, deliberadamente `Int`/arredondado por segundo "pra não atualizar demais" — conveniente aqui, é exatamente a granularidade certa pra debounce de thumbnail). O plano: um popup flutuante (imagem + timestamp) ancorado acima do thumb do `Slider`, visível só enquanto `onEditingChanged(true)` está ativo (linha 570), alimentado por um `Task` com debounce (~100-150ms) sobre mudanças de `model.currentTime`, chamando o novo método do protocolo e populando um cache em memória (chave = segundo arredondado, LRU pequeno) para não redecodificar o mesmo ponto duas vezes.

4. **Modelo de interação no remote da Apple TV é a maior incógnita de design, não de engenharia** — ver seção de Riscos.

## Arquivos e tipos afetados

- `Sources/KSPlayer/AVPlayer/MediaPlayerProtocol.swift:96` — adicionar ao protocolo `MediaPlayerProtocol` um novo método público (ex.: `func thumbnailImage(at time: TimeInterval) async -> CGImage?`), mantendo `thumbnailImageAtCurrentTime()` intocado (é usado por `IOSVideoPlayerView.swift:234` para outro propósito).
- `Sources/KSPlayer/AVPlayer/KSAVPlayer.swift:388-397,573-591` — implementar o novo método reaproveitando `createImageGenerator()`/`thumbnailImage(currentTime:handler:)`; considerar migrar para `AVAssetImageGenerator.image(at:)` (async, iOS/tvOS 16+) e ajustar tolerâncias para scrub.
- `Sources/KSPlayer/MEPlayer/KSMEPlayer.swift:438-440` — implementar o novo método delegando a um novo componente de thumbnail sob demanda (ver próximo item), em vez de só expor o `pixelBuffer` atual.
- `Sources/KSPlayer/MEPlayer/ThumbnailController.swift:24-132` — generalizar (ou criar um tipo companheiro, ex. `SeekThumbnailGenerator`) para: (a) aceitar `KSOptions`/`formatContextOptions` na abertura do `AVFormatContext` secundário; (b) manter esse contexto vivo entre chamadas em vez de abrir/fechar por request; (c) expor um método de busca pontual por tempo, com cancelamento cooperativo do request anterior (mesma lógica de "seek coalescente" já documentada para o demux principal em `MEPlayerItem.swift:690-692`); (d) rodar numa fila serial dedicada (mirror do padrão `OperationQueue` de `MEPlayerItem`, `docs/03-engine-meplayer-demux-e-pipeline.md`) para não colidir com o pipeline de decodificação ao vivo nem com dois seeks concorrentes no mesmo `AVFormatContext`.
- `Sources/KSPlayer/AVPlayer/KSOptions.swift` — novas flags públicas (nenhuma existe hoje: `rg -i "thumbnail|preview"` não retorna nada em `KSOptions.swift`), por exemplo `enableScrubThumbnails: Bool`, `scrubThumbnailWidth: Int32`, `scrubThumbnailDebounce: TimeInterval`, seguindo o padrão existente de knobs (`isAccurateSeek`, `nobuffer`, etc., linhas 20-91).
- `Sources/KSPlayer/AVPlayer/KSVideoPlayer.swift:330-336` (`ControllerTimeModel`) — adicionar estado observável para a imagem de preview atual (ex.: `@Published var previewImage: CGImage?`) ou um `ObservableObject` companheiro dedicado, para não sobrecarregar o modelo existente.
- `Sources/KSPlayer/SwiftUI/KSVideoPlayerView.swift:554-587` (`VideoTimeShowView`) — nova view de popup de thumbnail ancorada ao `Slider`, gatilho de debounce ligado a `onEditingChanged`/mudança de `model.currentTime`.
- Fora do escopo primário, mas relevante para paridade cross-platform do próprio fork (não é o caminho usado pelo StreamHub/tvOS): `Sources/KSPlayer/Core/UIKitExtend.swift:10-95` (`KSSlider`) já expõe um hook de arrasto real via `KSSliderDelegate.slider(value:event:.valueChanged)` (linhas 52-55) que o `PlayerToolBar`/`VideoPlayerView` (UIKit clássico, iOS/macOS) poderiam plugar sem trabalho extra de "detectar arrasto" — só falta a view de popup. O `UXSlider` do tvOS nesse caminho legado (`UIKitExtend.swift:94+`) é um `UIProgressView` fake sem interação nenhuma e não é o que o StreamHub usa (StreamHub usa o caminho SwiftUI acima).

## Dependências (de outras features/upgrades)

- **Nenhuma dependência bloqueante** — a feature pode ser construída hoje, em cima do `KSMEPlayer`/FFmpeg 6.1 atual, sem esperar o upgrade para FFmpeg 8.x nem o remux HLS local → AVPlayer.
- **Sobreposição futura com o remux HLS local → AVPlayer para Dolby Vision** (`context/investigation/native-dolby-vision-dynamic-metadata-p5-p8-p7-single-layer.md`, `context/investigation/proavplayer-mkv-com-dolby-vision-e-atmos-nativos-via-avplaye.md`): quando esse pipeline virar o caminho padrão para conteúdo DV-compatível, esses arquivos passam a rodar em `KSAVPlayer`, não `KSMEPlayer` — e o preview desses títulos usaria o caminho 1 (AVAssetImageGenerator), tornando o esforço do caminho 2 (FFmpeg sob demanda) parcialmente redundante só para essa fatia de conteúdo. Como hoje **todo** MKV passa por `KSMEPlayer`, os dois caminhos precisam existir de qualquer forma — mas vale sequenciar/orçar sabendo que o caminho AVPlayer tende a ganhar mais catálogo com o tempo.
- Nenhuma dependência de `Precache data to Hard Drive` (ausente, sem doc de roadmap próprio ainda) nem de `Video switching with zero delay`, embora as três compartilhem a mesma preocupação de fundo (custo de I/O extra contra um endpoint HTTP de debrid).

## Riscos e incógnitas

- **Custo de rede contra o endpoint de debrid**: cada seek de preview no engine FFmpeg é, na prática, uma ou mais requisições HTTP range novas contra a mesma URL que já está sendo streamada para playback — diferente do caso local/arquivo em que `av_seek_frame` é essencialmente grátis. Debrid endpoints frequentemente têm limite de conexões concorrentes, rate limit ou tokens de sessão de vida curta; abrir um segundo `AVFormatContext` sobre a mesma URL pode: (a) contar contra o limite de conexões e derrubar o stream principal; (b) sofrer latência alta o suficiente para inviabilizar preview "ao vivo" durante o arrasto. Precisa de teste real contra o(s) provedor(es) de debrid do StreamHub antes de comprometer a UX a isso — não há como confirmar isso só lendo código.
- **`ThumbnailController` abre sem headers/opções hoje** (`ThumbnailController.swift:49`, `nil` no lugar de `formatContextOptions`) — contra qualquer stream que exija header/token (praticamente todo endpoint de debrid do StreamHub), a implementação atual falharia silenciosamente. Confirmado por comparação direta com `MEPlayerItem.openThread` (`MEPlayerItem.swift:191,202`), que passa `options.formatContextOptions.avOptions`. É um requisito, não um nice-to-have.
- **Modelo de interação no remote da Apple TV é a maior incógnita, não a decodificação**: a pesquisa de mercado (ver Referências) sugere que a própria Infuse, em apps de tvOS sem touch surface ativo, usa o padrão "pausar → mover o foco/seta esquerda-direita → thumbnail atualiza a cada passo → confirmar" em vez de "arrastar um slider continuamente". O fork hoje já tem os dois ganchos de remote (`onMoveCommand`/`onKeyPressLeftArrow`/`onKeyPressRightArrow` em `KSVideoPlayerView.swift:150-210`, hoje usados só para skip de 15s) e o `Slider` do SwiftUI (`VideoTimeShowView`, que em teoria atualiza `currentTime` continuamente enquanto o usuário desliza o dedo na superfície touch do Siri Remote, mas na prática o comportamento de foco+swipe de `Slider` em tvOS é conhecido por ser impreciso/instável em relatos de terceiros). Decidir entre "continuous drag preview" (mexe pouco na UI atual) e "discrete step preview" (mais fiel ao padrão Infuse/tvOS nativo, mas exige repensar a interação do scrub bar) é uma decisão de produto que muda a estimativa de esforço da camada de UI.
- **Concorrência sobre o `AVFormatContext` secundário**: se o usuário arrasta rápido, múltiplos seeks pontuais podem ser disparados em sequência; sem coalescer (mesmo padrão documentado em `docs/03-engine-meplayer-demux-e-pipeline.md` para o seek do demux principal), o segundo contexto FFmpeg pode enfileirar seeks obsoletos e devolver uma prévia atrasada em relação a onde o usuário já está.
- **Frame na keyframe ≠ frame exato do tempo pedido**: como no `ThumbnailController` atual, `AVSEEK_FLAG_BACKWARD` + 1 decode devolve o frame do keyframe anterior, não o frame exato do tempo alvo — aceitável (é o mesmo trade-off que Jellyfin/mpv fazem de propósito por performance), mas pode gerar prévias visivelmente "atrasadas" em relação ao tempo mostrado, especialmente em conteúdo com GOPs longos (comum em encodes 4K remux/WEB-DL).
- **Anime com legendas ASS embutidas** (perfil de conteúdo citado como prioritário pelo dono do projeto): thumbnails não devem (e não precisam) renderizar legenda queimada — o pipeline do `ThumbnailController`/`VideoSwresample` já opera só sobre o frame de vídeo decodificado, sem o subtitle overlay da UI, então não há risco aqui, mas vale confirmar que nenhuma escolha futura de implementação acople os dois caminhos.
- **HDR/Dolby Vision no próprio frame de preview**: o frame gerado sob demanda vem de um contexto de decode "cru" (sem `VideoToolboxDecode`, sem os shaders EDR de `docs/06-render-de-v-deo-e-hdr.md`) — para conteúdo P5 Dolby Vision (RPU only, sem base layer SDR embutida) a imagem pode sair com cores erradas/lavadas se renderizada sem tone-mapping; precisa de um caminho de conversão de cor mínimo (ou aceitar a limitação e documentar) — não investigado a fundo aqui, é um ponto em aberto.
