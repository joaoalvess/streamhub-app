**Dificuldade:** L · **Relevância:** crítica

## Abordagem proposta

O achado central: esta feature **não se implementa dentro deste repositório**. `StreamHub/Player` só consome um pin de versão (`Package.swift:46`); todo o código-fonte do FFmpeg e o script que o compila para Apple platforms vivem no pacote irmão `kingslay/FFmpegKit`. Subir de 6.1.x para 8.x é, portanto, um trabalho em duas frentes bem separadas:

**Frente 1 — no repositório externo `kingslay/FFmpegKit` (ou num fork dele).** O pin real da versão do FFmpeg está em `Plugins/BuildFFmpeg/main.swift`, no enum `Library`: `case .FFmpeg: return "n6.1"` — uma tag git do mirror oficial `https://github.com/FFmpeg/FFmpeg` (a URL cai no `default:` do switch de `url`, que monta `https://github.com/FFmpeg/FFmpeg` a partir do `rawValue`). Confirmei via `git ls-remote --tags` que esse mirror já tem as tags `n8.0`, `n8.0.1`, `n8.0.2`, `n8.0.3`, `n8.1`, `n8.1.1` e `n8.1.2` — ou seja, a versão-alvo que o `README.md:64` deste repo já anuncia para a versão paga (8.1.1) existe e é buildável hoje. O procedimento mecânico seria: trocar essa string por `"n8.1.1"` (ou `"n8.1.2"`, o mais recente) e rodar `swift package --disable-sandbox BuildFFmpeg` no clone do FFmpegKit — comando que faz cross-compile de FFmpeg + ~20 bibliotecas dependentes (libdav1d, libplacebo, libass, MoltenVK, libshaderc, gmp/nettle/gnutls, libsrt, libsmbclient, libbluray…) para cada plataforma declarada (macOS, iOS, iOS-simulator, tvOS, tvOS-simulator, xrOS, xrOS-simulator, maccatalyst) e monta os `.xcframework` via `xcodebuild -create-xcframework`. Isso roda localmente (exige Xcode completo + Homebrew: `nasm`, `sdl2`, `pkg-config`, `cmake`) e não existe em CI — é um processo manual, de horas, no Mac de quem fizer o bump.

Confirmei também que **não existe hoje** um fork do FFmpegKit sob a conta do dono (`gh api repos/joaoalvess/FFmpegKit` → 404; nenhum fork listado nos forks públicos do upstream) — quem fizer esse trabalho começa do zero, seja enviando PR para o `kingslay/FFmpegKit` upstream (repo ativo — `pushed_at` recente, dentro das últimas duas semanas) ou criando um fork próprio. Dado que o objetivo final é uso pessoal no StreamHub (não redistribuição), um **fork próprio é o caminho pragmático default**; abrir PR upstream é um bônus se o bump sair limpo e sobrar tempo, mas o maintainer controla timeline/QA e pode preferir não mexer (uma versão por vez, estabilidade).

**Frente 2 — neste repositório, depois que a Frente 1 produzir binários.** Trocar `Package.swift:46` para apontar para a tag/branch nova do fork, deixar o `Package.resolved` se regenerar, e então **compilar e seguir os erros** em `Sources/KSPlayer/MEPlayer/*.swift` — não há uma lista fechada de breakages conhecidos hoje porque ninguém tentou esse bump ainda neste código; a validação é empírica.

Boa notícia levantada pela investigação de código: o blast radius nesta Frente 2 parece menor do que o salto de versão sugere à primeira vista. `AVFFmpegExtension.swift`, `Resample.swift` e `AVFoundationExtension.swift` já usam a API moderna `AVChannelLayout`/`av_channel_layout_*` (não o par legado `channels`/`channel_layout` de `AVCodecContext`, removido em FFmpeg 7.0) — a maior baixa de compatibilidade binária/fonte da era 5.x→7.x já foi absorvida. Não há chamadas a APIs há muito removidas (`avcodec_decode_video2`, `av_register_all`); `avformat_network_init`, `AVFMT_FLAG_GENPTS`/`AVFMT_FLAG_NOBUFFER`/`AVFMT_FLAG_CUSTOM_IO` e `sws_getContext` (comentado, não usado) seguem válidos nos headers do FFmpeg 8.x. Ou seja: o código Swift deste fork já está estruturalmente mais perto de FFmpeg 8.x do que de FFmpeg 5.x — o que não elimina risco, só o reduz.

O risco maior e menos previsível está mesmo na Frente 1: a receita de build do FFmpegKit depende de três coisas frágeis entre majors do FFmpeg — (a) um patch textual em `libavcodec/videotoolbox.c` (troca de `kCVPixelBufferOpenGLESCompatibilityKey`/`kCVPixelBufferIOSurfaceOpenGLTextureCompatibilityKey` por `kCVPixelBufferMetalCompatibilityKey`) cujo texto-alvo pode ter mudado de contexto no refactor do hwaccel VideoToolbox entre 6.1 e 8.x; (b) uma lista de headers **internos** do FFmpeg copiados manualmente para dentro do xcframework (`getenv_utf8.h`, `libm.h`, `thread.h`, `intmath.h`, `mem_internal.h`, `attributes_internal.h`, `mathops.h`, `os_support.h`) — API não estável, sem garantia de existirem com o mesmo nome/local em 8.x; (c) uma lista extensa e cuidadosamente escolhida de flags `--enable-decoder=`/`--enable-demuxer=`/`--enable-muxer=`/`--enable-filter=` que precisa continuar resolvendo contra o `configure` do FFmpeg 8.x (nomes/dependências de componentes mudam entre majors).

- `Package.swift` do FFmpegKit — sem mudança estrutural esperada; os `binaryTarget` continuam apontando para os mesmos `path:`, só o conteúdo dos `.xcframework` muda.

## Dependências

- **Nenhuma dependência bloqueante de outra feature do roadmap para começar.** O doc de HDR10+ (`context/investigation/hdr10-dynamic-metadata.md` / `context/roadmap/hdr10-dynamic-metadata.md`) já registra que `hdr_dynamic_metadata.h`/`av_dovi_get_header` etc. compilam e funcionam na 6.1.4 atual — nada no roadmap de qualidade (HDR10+, Dolby Vision P5/P8/P7, Atmos) está tecnicamente travado esperando este bump.
- **É pré-requisito estratégico (não bloqueante) das outras "joias da coroa".** FFmpeg 8.x traz correções de demux/decode mais recentes (HEVC, AV1/dav1d, MKV) e filtros de tone-mapping mais atuais que só ajudam indiretamente HDR10+/Dolby Vision/Atmos — mas nenhuma delas está impedida de avançar com 6.1.4 hoje. Vale sequenciar o bump quando houver folga, não antes de features com dependência técnica real.
- **Depende de infraestrutura de build fora deste repo**: máquina macOS com Xcode completo + Homebrew (`nasm`, `sdl2`, `pkg-config`, `cmake`) e horas de CPU livre para rodar (e re-rodar, até compilar limpo) `swift package --disable-sandbox BuildFFmpeg` em todas as plataformas declaradas.
- **Depende de uma decisão de propriedade**: criar `joaoalvess/FFmpegKit` (hoje inexistente) vs. tentar PR upstream no `kingslay/FFmpegKit` ativo.

## Riscos e incógnitas

- **Headers internos copiados à mão são o ponto mais frágil.** `internal.h`, `mem_internal.h`, `attributes_internal.h`, `getenv_utf8.h`, `thread.h`, `intmath.h`, `mathops.h`, `os_support.h` são API não pública do FFmpeg — não há garantia de que existam com o mesmo nome/conteúdo em 8.x. Se algum desapareceu ou mudou de diretório, o script de build quebra e exige investigação caso a caso (nem o autor do FFmpegKit necessariamente já resolveu isso, já que o upstream segue travado em 6.1.4).
- **O patch textual em `videotoolbox.c` pode falhar silenciosamente.** É uma substituição de string (`replacingOccurrences`); se o texto-alvo não existir mais (o hwaccel VideoToolbox do FFmpeg foi mexido mais de uma vez entre 6.1 e 8.x), o `try!` do patch pode não ser o ponto de falha visível — o build pode compilar e só quebrar em runtime no hardware decode.
- **Custo de build é medido em horas, não minutos**, multiplicando ~8 plataformas × arquiteturas × ~20 bibliotecas; historicamente sensível a falhas de rede/Homebrew/certificados. Não é algo que se valida numa sessão só, nem que tem CI própria — o workflow deste repo (`.github/workflows/build.yml`) só faz `swift build`/`swift test`, nunca recompila FFmpeg.
- **Sem rede de segurança automatizada para o resultado.** Os testes de mídia em `KSPlayerTests` são no-op silenciosos (fixtures ausentes, já documentado em `docs/01`) — qualquer regressão de decode introduzida pelo bump só aparece em teste manual com arquivos reais (os MKV 4K/anime que o StreamHub de fato consome).
- **`arthenica/ffmpeg-kit` não é atalho.** Confirmei via `gh api` que está arquivado (`archived: true`) e parou no FFmpeg 6.0 — reforça que recompilar FFmpeg para múltiplas plataformas Apple é trabalho genuinamente caro (um projeto mantido por anos e usado amplamente desistiu dele), não um problema já resolvido em algum lugar que dê para só importar.
- **Peso do repositório.** O FFmpegKit já é descrito como "clone multi-GB" por commitar xcframeworks binários (`docs/01`); manter um fork implica hospedar e sincronizar esse peso indefinidamente, e cada rebuild gera um novo conjunto de binários grandes para commitar.
- **Retorno não medido ainda.** Sem tentar, não dá para saber se o ganho real (decoders/bugfixes específicos do FFmpeg 8.x) supera o custo de manter um fork de build binário para sempre — vale checar, antes de investir, se algum decoder/formato do catálogo real do StreamHub (remuxes/WEB-DL 4K, anime com ASS) realmente precisa de algo que só existe a partir do 7.x/8.x, ou se o ganho é majoritariamente "número da versão bater com a tabela do README".
