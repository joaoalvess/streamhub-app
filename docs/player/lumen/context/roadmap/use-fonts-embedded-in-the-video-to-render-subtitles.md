**Dificuldade:** M · **Relevância:** alta

## Abordagem proposta

Ao contrário do HDR10+ dinâmico (onde falta API da Apple), aqui **todas as peças existem** — o problema é 100% de plumbing dentro deste fork. A abordagem de referência usada por VLC, mpv e Jellyfin é sempre a mesma, independente do motor de render de legenda: (1) extrair os anexos de fonte do contêiner, (2) entregá-los ao motor de renderização de texto antes do primeiro parse de estilo. O que muda por player é só o passo 2 — libass tem uma API dedicada (`ass_add_font`), este fork não tem libass ativo (`import libass`/`AssParse` é parser Swift próprio, `docs/00-ARQUITETURA.md:114`), então o passo 2 aqui é registrar as fontes no CoreText do sistema operacional e fazer `AssParse`/`String.parseStyle`/`parseASSStyle()` consultá-las.

**1) Extração dos anexos (`MEPlayerItem.createCodec`, `Sources/KSPlayer/MEPlayer/MEPlayerItem.swift:335-349`).** Hoje esse loop cria um `FFmpegAssetTrack` por stream de áudio/vídeo/legenda; `FFmpegAssetTrack.init?(codecpar:)` retorna `nil` para qualquer `codec_type` fora desse trio (`FFmpegAssetTrack.swift:144-245` só trata `AVMEDIA_TYPE_AUDIO/VIDEO/SUBTITLE`), então hoje um stream `AVMEDIA_TYPE_ATTACHMENT` de fonte é silenciosamente descartado pelo `compactMap`. A extração entra como um `else if` irmão do `if let assetTrack = FFmpegAssetTrack(stream: coreStream)` (linha 338): quando `coreStream.pointee.codecpar.pointee.codec_type == AVMEDIA_TYPE_ATTACHMENT`, ler `toDictionary(coreStream.pointee.metadata)` (helper já existente, `AVFFmpegExtension.swift:7-17`) para pegar `filename`/`mimetype`, filtrar por mimetype (`application/x-truetype-font`, `application/vnd.ms-opentype`, `font/ttf`, `font/otf`) ou, na ausência de mimetype útil, pela extensão do `filename` (`.ttf`/`.otf`/`.ttc`) — é exatamente o heurístico que o VLC usa (`modules/codec/libass.c:171-186`: checa mimetype `application/x-truetype-font` e cai para checar os últimos 4 caracteres do nome do arquivo). Os bytes ficam em `codecpar.extradata`/`extradata_size` (convenção padrão do FFmpeg para `AVMEDIA_TYPE_ATTACHMENT`, simétrica entre `matroskadec.c`/`matroskaenc.c`). Isso **não** vira `MediaPlayerTrack`/`FFmpegAssetTrack` — segue o mesmo padrão já usado para capítulos (`nb_chapters`/`chapters: [Chapter]`, `MEPlayerItem.swift:246-258`): um array simples de "coisas descobertas na abertura", sem exposição na UI de seleção de trilha.

**2) Registro no CoreText.** Para cada fonte extraída: `CGDataProvider(data: data as CFData)` → `CGFont(dataProvider)` → `CTFontManagerRegisterGraphicsFont(font, &error)`. Essa API (não a variante `RegisterFontsForURL`) foi escolhida porque aceita os bytes diretamente da memória — sem escrever em `NSTemporaryDirectory()` nem gerenciar arquivo temporário, e sem depender de closure de I/O. Ela registra no *scope* do processo (equivalente a instalado, mas some ao sair do processo); o desregistro simétrico é `CTFontManagerUnregisterGraphicsFont(font, &error)`.

**3) Tabela de resolução de nomes.** O ponto mais delicado: `UIFont(name:)` resolve por **PostScript name**, mas o campo `Fontname` do ASS (e o `\fn` inline) é tipicamente o **family name** como o autor do `.ass` o digitou — nem sempre coincidem (ex.: family "Noto Sans CJK JP" vs PostScript "NotoSansCJKjp-Regular"). Registrar a fonte no CoreText não resolve isso sozinho. É necessário, no momento do registro, extrair **as duas chaves** via `CTFontCreateWithGraphicsFont(font, 0, nil, nil)` + `CTFontCopyName(_:kCTFontFamilyNameKey)`/`kCTFontPostScriptNameKey`, normalizar (trim + case-insensitive, fansub ASS costuma variar capitalização) e alimentar uma tabela `[String: CTFont]` consultável pelos três pontos que hoje chamam `UIFont(name:)` cegamente:
   - `Sources/KSPlayer/Subtitle/KSParseProtocol.swift:242` (`\fn`/`\fs` inline, dentro de `String.parseStyle`)
   - `Sources/KSPlayer/Subtitle/KSParseProtocol.swift:253` (`Fontname`/`Fontsize` do estilo do cabeçalho, `[String: String].parseASSStyle()`)
   - `Sources/KSPlayer/Subtitle/KSParseProtocol.swift:261-262` (`UIFontDescriptor(name:matrix:)` do mesmo método, para estilos com `Angle`)
   
   O fallback para `UIFont(name:) ?? UIFont.systemFont(ofSize:)` continua existindo (mantém o comportamento atual quando não há fonte embutida ou quando ela não bate com nada), mas passa a ser o *segundo* passo, não o único.

**4) Ciclo de vida.** Registro síncrono dentro de `createCodec` (chamado de `openThread`, `MEPlayerItem.swift:245`, antes de `read()` iniciar) garante que qualquer `SubtitleDecode`/`AssParse` que rodar depois (sempre em uma OperationQueue por trilha, criada só depois de `createCodec` retornar) já encontre as fontes registradas — não há race de "parser correu antes do registro" no caminho feliz. O desregistro entra no bloco de `shutdown()` que já fecha o `formatCtx` (`MEPlayerItem.swift:628-654`, especificamente perto da linha 647 `avformat_close_input(&self.formatCtx)`).

**Onde a tabela de nomes deve morar**: `Sources/KSPlayer/Subtitle/` importa só Foundation/UIKit/SwiftUI (nunca FFmpeg), enquanto a extração dos bytes vive em `Sources/KSPlayer/MEPlayer/` (importa `Libavformat`). Para não vazar tipos do FFmpeg para a camada de parsing, a tabela de nomes deve ser um tipo novo e simples (`Data`/`String` apenas) dentro de `Subtitle/`, populado pelo `MEPlayerItem` via uma chamada com tipos Swift puros — mesmo padrão de fronteira já usado por `KSOptions.subtitleParses`/`SubtitleModel.audioRecognizes` (estáticos em `Subtitle/` alimentados de fora).

**Toggle em `KSOptions`**: seguindo o padrão existente (`KSOptions.autoSelectEmbedSubtitle`, `KSOptions.isSeekImageSubtitle`, `KSOptions.swift:74-75`), expor `KSOptions.registerEmbeddedFonts: Bool = true` como o painel de controle desta feature — permite desligar sem tocar no pipeline.

## Arquivos e tipos afetados

- `Sources/KSPlayer/MEPlayer/MEPlayerItem.swift:335-349` — novo ramo no loop de `createCodec` para `AVMEDIA_TYPE_ATTACHMENT`; extração de `filename`/`mimetype` via `toDictionary(coreStream.pointee.metadata)` e bytes via `codecpar.extradata`/`extradata_size`.
- `Sources/KSPlayer/MEPlayer/MEPlayerItem.swift:628-654` (`shutdown`) — desregistro simétrico das fontes no fechamento, ao lado de `avformat_close_input`.
- `Sources/KSPlayer/MEPlayer/FFmpegAssetTrack.swift:144-245` (`init?(codecpar:)`) — nenhuma mudança necessária; serve só de referência de que anexos já são filtrados fora do conceito de `MediaPlayerTrack` hoje (comportamento a preservar).
- `Sources/KSPlayer/MEPlayer/AVFFmpegExtension.swift:7-17` (`toDictionary`) — reuso direto, sem alteração.
- **Novo tipo** (sugestão: `Sources/KSPlayer/Subtitle/EmbeddedFontRegistry.swift`) — tabela `[String: CTFont]` (chaves: family name e PostScript name normalizados), API mínima `register(data:) `/`unregisterAll()`/`font(named:size:) -> UIFont?`; tipo `Sendable`-friendly (CoreText/CGFont/CTFont não são `Sendable` por padrão sob `StrictConcurrency`, `Package.swift:30-32` — precisa de wrapper `@unchecked Sendable` ou de rodar só dentro de uma fila serial dedicada, seguindo a disciplina já documentada em `docs/00-ARQUITETURA.md` item 7/12).
- `Sources/KSPlayer/Subtitle/KSParseProtocol.swift:242,253,261-262` — os três pontos de `UIFont(name:)`/`UIFontDescriptor(name:matrix:)` passam a consultar o registry antes do fallback atual.
- `Sources/KSPlayer/AVPlayer/KSOptions.swift` (próximo de `:74-75`) — novo `static var registerEmbeddedFonts = true`.
- **Bloqueador funcional fora do escopo literal do anexo, mas inseparável na prática**: `Sources/KSPlayer/Subtitle/KSSubtitle.swift:346-351` (`SubtitleModel.subtitle(currentTime:)`). Hoje, toda vez que o conjunto de `parts` visível muda (ou seja, a cada nova fala/evento ASS — não é um caso raro, é o caminho comum), o código roda `text.addAttributes([.font: SubtitleModel.textFont], range: NSRange(location: 0, length: text.length))` sobre **qualquer** `NSMutableAttributedString`, sobrescrevendo incondicionalmente a chave `.font` com a fonte global (system font bold/não-bold) — isto é o mesmo mecanismo que já anula hoje `\fn`/`Fontname` normais (documentado como pegadinha em `docs/07-legendas.md:103`). **Sem alterar essa linha, o registro/resolução de fonte embutida seria código morto**: o `.font` correto seria montado em `String.parseStyle`/`parseASSStyle()` e imediatamente substituído antes da primeira exibição. Correção mínima: só aplicar `SubtitleModel.textFont` em ranges que **não** têm `.font` já setado (ex.: `text.enumerateAttribute(.font, in: fullRange) { value, range, _ in if value == nil { text.addAttribute(.font, value: SubtitleModel.textFont, range: range) } }`), preservando fontes por-run (ASS ou embutida) e usando a global só como piso para texto sem estilo.

## Dependências

- **Nenhuma dependência do upgrade FFmpeg 6.1→8.x**: a convenção `AVMEDIA_TYPE_ATTACHMENT`/`extradata`/`extradata_size` para anexos do Matroska é estável há mais de uma década no FFmpeg; nada aqui precisa de API nova do libavformat.
- **Dependência forte e não-opcional em `docs/07-legendas.md` pegadinha "Fonte global sobrescreve fonte ASS"** (`KSSubtitle.swift:346-351`): ver bloqueador funcional acima — esta é uma dependência de fato, não um risco lateral. Recomenda-se implementar essa correção **como parte do mesmo trabalho**, não como item separado, porque isoladamente nenhuma das duas metades produz efeito visível.
- **Sobreposição de infraestrutura com "Full ASS subtitle effects (render via libass)"** (`docs/README.md:79`, sem investigação própria hoje): se o fork decidir reativar o produto `libass` do FFmpegKit (comentado em `Package.swift:25`, ver `docs/01-vis-o-geral-e-build.md:56`), a extração de anexos do passo 1 é reaproveitada **sem alteração** — só troca o consumidor do passo 2 (`ass_add_font(library, name, data, size)` em vez de `CTFontManagerRegisterGraphicsFont`), e o passo 3 (tabela de nomes) deixa de ser necessário porque o fontselect interno do libass (com backend CoreText nativo em macOS/iOS, sem precisar de fontconfig) já resolve family name ⇄ PostScript name sozinho. Ou seja: implementar isto agora não é trabalho descartável se a migração para libass acontecer depois — é a metade que sobrevive.
- **Nenhuma dependência de "MKV com Dolby Vision/Atmos nativos via AVPlayer"**: esse remux alimenta o `KSAVPlayer`, cujo `subtitleDataSouce` é sempre `nil` (`KSAVPlayer.swift:363`, `docs/07-legendas.md:45`) — legendas embutidas (e portanto fontes embutidas) só existem no caminho `KSMEPlayer`/FFmpeg hoje. Conteúdo que migrar para o pipeline `AVPlayer` nativo perde acesso a esta feature (mas também perde acesso a toda legenda embutida, não é uma regressão nova).

## Riscos e incógnitas

- **Mismatch family name vs PostScript name é o risco central**, não um detalhe de implementação: mesmo com o registro CoreText funcionando perfeitamente, se a tabela de resolução não indexar por **ambos** os nomes (normalizados), boa parte das fontes embutidas continuará caindo no fallback do sistema — silenciosamente, como hoje. Esse é o mesmo problema relatado em `xbmc/xbmc#16209` ("External ASS subtitles with custom fonts always display with default font") e discutido em `libass/libass#535`.
- **Fontes com nome ofuscado/anti-leech**: alguns grupos de fansub embutem fontes sob um family name que não corresponde a nada usado no `.ass` (prática antiga de "proteção" de fonte) — nenhuma implementação corrige isso, é um problema de autoria upstream. Vale não perseguir esse caso como bug do fork.
- **Registro duplicado entre trocas de mídia**: `CTFontManagerRegisterGraphicsFont` falha se o PostScript name já está registrado no processo. Dois arquivos consecutivos que embutem fontes com o mesmo PostScript name (comum: mesma release/fonte de fansub reaproveitada entre episódios de uma série) podem colidir se o desregistro do arquivo anterior não tiver terminado antes do registro do novo — relevante porque "troca de stream sem delay" é prioridade explícita do dono; a implementação precisa tolerar erro de "já registrado" (ignorar, reaproveitar o já registrado) em vez de propagar falha, e/ou versionar o registro por instância de `MEPlayerItem` para desregistrar exatamente o que aquela instância registrou.
- **`StrictConcurrency`**: `CGFont`/`CTFont`/`Unmanaged<CFError>` não são `Sendable`; o registro acontece na OperationQueue de abertura (`openThread`) e a leitura acontece na OperationQueue de decodificação da trilha de legenda (thread diferente, por trilha) — precisa do mesmo tratamento manual de isolamento já usado no resto do `MEPlayerItem` (`docs/00-ARQUITETURA.md` itens 7 e 12: tipos novos devem ser `nonisolated`/`Sendable` desde o início, sem depender do compilador para pegar erro).
- **Sem cobertura de teste automatizada**: a suíte XCTest do pacote é no-op por fixtures ausentes (`docs/09-demo-e-testes.md`, citado em `docs/00-ARQUITETURA.md` item 12) — validação será manual, com um MKV real com fonte embutida + ASS referenciando-a (comum em remuxes/WEB-DL de anime, o público-alvo do StreamHub).
- **Filtragem de mimetype/extensão pode ter falso positivo/negativo**: anexos MKV nem sempre são fontes (capa de álbum, thumbnails, `.nfo`); o heurístico do VLC (mimetype exato ou 4 últimos caracteres do nome) é uma aproximação testada em produção, não uma garantia — arquivos com mimetype ausente/genérico (`application/octet-stream`) e extensão atípica escapam da detecção, mesma limitação que o VLC aceita.
- **Efeito visual pode ser sutil ou nulo sem o fix do overwrite global**: reforçando a dependência acima — se a correção de `KSSubtitle.swift:346-351` for adiada, o time de QA (o próprio dono validando manualmente) veria "nenhuma mudança visível" mesmo com a extração/registro funcionando corretamente, o que pode ser mal-diagnosticado como falha da feature em si.
