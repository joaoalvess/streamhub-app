## Status

Parcial.

Existem hooks reais de extração de metadados Dolby Vision (config record no bitstream e side-data de RPU/metadata por frame vindos do FFmpeg), mas os dados extraídos são descartados — não há profile-awareness (P5 vs P7/P8) nem aplicação dos metadados dinâmicos (RPU) à cadeia de renderização. O caminho de exibição real de Dolby Vision depende inteiramente do passthrough nativo do VideoToolbox/AVSampleBufferDisplayLayer via `formatDescription`, não de uma implementação própria de dynamic metadata.

## Evidência

- `Sources/KSPlayer/AVPlayer/MediaPlayerProtocol.swift:153-162` — `struct DOVIDecoderConfigurationRecord` com `dv_profile`, `dv_level`, `rpu_present_flag`, `el_present_flag`, `bl_present_flag`, `dv_bl_signal_compatibility_id` — todos os campos que distinguiriam P5 (single-layer, sem base layer) de P7/P8 (dual/single layer com base compatível) estão modelados no struct.
- `Sources/KSPlayer/MEPlayer/FFmpegAssetTrack.swift:167-168` — o `AV_PKT_DATA_DOVI_CONF` do lado do pacote é lido e reinterpretado em `DOVIDecoderConfigurationRecord`, guardado em `FFmpegAssetTrack.dovi` (`FFmpegAssetTrack.swift:37`).
- Busca em todo o repo (`rg -n "dv_profile"`) mostra que `dv_profile` só aparece na declaração do struct — nenhum branch de código lê esse campo para diferenciar P5/P7/P8. `dovi` é usado apenas como `!= nil` (booleano `isDovi`) em `FFmpegDecode.swift:31`, `VideoToolboxDecode.swift:65`, `MetalPlayView.swift:33/36/51/180`, `Model.swift:428-433` — serve só para sinalizar "tem Dolby Vision" e ajustar refresh rate / criação de `VideoVTBFrame`/`VideoSwresample`, não para tratar os perfis diferentemente.
- `Sources/KSPlayer/MEPlayer/FFmpegDecode.swift:94-95` — `AV_FRAME_DATA_DOVI_RPU_BUFFER`: o buffer RPU é lido para uma variável local `data` e **nunca usado** (sem warning suprimido, sem atribuição a frame/pixel buffer).
- `Sources/KSPlayer/MEPlayer/FFmpegDecode.swift:96-101` — `AV_FRAME_DATA_DOVI_METADATA` (`AVDOVIMetadata`): `header`, `mapping` e `color` são extraídos via `av_dovi_get_header/mapping/color`, mas a única linha que faria algo com isso está comentada: `// frame.corePixelBuffer?.transferFunction = kCVImageBufferTransferFunction_ITU_R_2020` (linha 101). Os valores extraídos não alimentam `EDRMetaData` nem qualquer outro estado do frame.
- Contraste: os side-data de HDR10 estático (`AV_FRAME_DATA_MASTERING_DISPLAY_METADATA`, `AV_FRAME_DATA_CONTENT_LIGHT_LEVEL`, `AV_FRAME_DATA_AMBIENT_VIEWING_ENVIRONMENT`) no mesmo laço (`FFmpegDecode.swift:106-133`) **são** de fato usados: convertidos em `MasteringDisplayMetadata`/`ContentLightMetadata`/`AmbientViewingEnvironment` e atribuídos a `videoFrame.edrMetaData` em `FFmpegDecode.swift:144-146`. HDR10+ dinâmico (`AV_FRAME_DATA_DYNAMIC_HDR_PLUS`, linha 102-103) e HDR Vivid (linha 104-105) seguem o mesmo padrão do Dolby Vision: extraídos e descartados.
- `Sources/KSPlayer/AVPlayer/PlayerDefines.swift:45-135` — `DynamicRange` trata `.dolbyVision` como um único caso monolítico (mapeado 1:1 para `AVPlayer.HDRMode.dolbyVision`), sem sub-distinção de perfil; `transferFunction` para `.dolbyVision` sempre retorna HLG (linha 124), nunca PQ, então não há tratamento por perfil na escolha de transfer function.
- `Sources/KSPlayer/AVPlayer/KSOptions.swift:339-351` — `updateVideo(refreshRate:isDovi:formatDescription:)`: quando `dynamicRange == .dolbyVision`, o único efeito é repassar o `formatDescription` (produzido pelo VideoToolbox/AVFoundation) para a camada de exibição — ou seja, a apresentação real de Dolby Vision (incluindo os metadados dinâmicos que o SDK da Apple aplica) é delegada ao decode/composição nativos do sistema, não implementada pelo KSPlayer.
- `Sources/KSPlayer/AVPlayer/KSAVPlayer.swift:527` — replica o mesmo campo `dovi: DOVIDecoderConfigurationRecord?` no player baseado em `AVPlayer`, também não lido em nenhum outro ponto desse arquivo.

## Como funciona (o que existe)

1. Ao abrir a track de vídeo, `FFmpegAssetTrack` varre os side-data do pacote (`AV_PKT_DATA_DOVI_CONF`) e, se presente, popula `dovi: DOVIDecoderConfigurationRecord` com os campos brutos do bitstream (perfil, nível, flags de RPU/EL/BL).
2. Esse valor só é consultado como booleano (`assetTrack.dovi != nil`) para setar `isDovi` em `VideoSwresample`/`VideoVTBFrame`, que por sua vez alimenta `KSOptions.updateVideo(refreshRate:isDovi:formatDescription:)` — usado apenas para lógica de refresh rate/HDR mode e para repassar o `formatDescription` do VideoToolbox à camada `AVSampleBufferDisplayLayer`.
3. Durante a decodificação por frame em `FFmpegDecode.decodeFrame`, o laço de side-data por frame lê `AV_FRAME_DATA_DOVI_RPU_BUFFER` e `AV_FRAME_DATA_DOVI_METADATA` (RPU parseado pelo próprio libavcodec/libavutil em `AVDOVIMetadata`, com header/mapping/color decompostos), mas os resultados não saem do escopo do `if`/`else if` — não são anexados a `EDRMetaData`, não influenciam `transferFunction`/`colorPrimaries`, não chegam ao Metal renderer (`MetalPlayView`/`PixelBufferProtocol`).
4. A exibição de Dolby Vision que de fato acontece hoje depende do pipeline nativo da Apple: o VideoToolbox decodifica o stream (incluindo, quando suportado por hardware, perfis DV compatíveis) e produz um `CMFormatDescription`/`CVPixelBuffer` com os atributos DV corretos; o KSPlayer apenas repassa esse `formatDescription` para a camada de composição. Isso cobre perfis onde o SO/hardware já faz o trabalho (ex.: profile 8.1 com base HDR10, comum em conteúdo de streaming), mas não é "dynamic metadata" implementado pelo player — é passthrough do sistema.

## O que falta

Para reivindicar "Dolby Vision dynamic metadata P5/P8/P7 single-layer" de fato, faltaria:

1. **Profile-awareness real**: ler `assetTrack.dovi.dv_profile`/`dv_level`/`bl_present_flag`/`el_present_flag` (já existem no struct, `MediaPlayerProtocol.swift:153-162`) para diferenciar P5 (single-layer BL, sem compatibilidade HDR10 — `bl_present_flag`=1, `el_present_flag`=0, sinal IPT-PQc2) de P7 (dual-layer, MEL/FEL) de P8 (single-layer com base HDR10/SDR compatível, `dv_bl_signal_compatibility_id` relevante). Hoje `DynamicRange.dolbyVision` é um caso único sem essa distinção (`PlayerDefines.swift:45-135`).
2. **Aplicar o RPU/metadata decodificado**: em `FFmpegDecode.swift:94-101`, usar `header`/`mapping`/`color` (já extraídos de `AVDOVIMetadata`) para de fato compor uma curva de mapeamento tonal por frame (ou, no mínimo, popular `EDRMetaData`/um novo `DoviMetaData` análogo ao `MasteringDisplayMetadata` existente) e propagar isso até `VideoVTBFrame`/`PixelBuffer` e o pipeline Metal (`MetalPlayView.swift`, `PixelBufferProtocol.swift`) para influenciar transfer function/tone mapping por frame — replicando o padrão que já existe para HDR10 estático (`FFmpegDecode.swift:144-146` → `videoFrame.edrMetaData`).
3. **Fallback via software para quando não há passthrough nativo** (ex.: macOS sem tela DV, ou decodificação puramente por software/FFmpeg sem VideoToolbox): usar o RPU (`AV_FRAME_DATA_DOVI_RPU_BUFFER`, hoje descartado em `FFmpegDecode.swift:95`) para aplicar tone-mapping dinâmico via shader Metal, já que sem esse trabalho o conteúdo P5 cairia para SDR/HDR10 estático sem os ajustes dinâmicos de brilho por cena/frame que o Dolby Vision promete.
4. Arquivos/tipos que seriam tocados numa implementação: `Sources/KSPlayer/MEPlayer/FFmpegDecode.swift` (extrair e propagar), um novo tipo `DoviMetaData`/extensão de `EDRMetaData` em `Sources/KSPlayer/MEPlayer/Model.swift`, `Sources/KSPlayer/AVPlayer/PlayerDefines.swift` (subdividir `DynamicRange.dolbyVision` por perfil ou adicionar metadado associado), `Sources/KSPlayer/Metal/PixelBufferProtocol.swift` e `Sources/KSPlayer/MEPlayer/MetalPlayView.swift` (consumir o metadata dinâmico no shader/composição), e possivelmente `Sources/KSPlayer/AVPlayer/KSOptions.swift` (`updateVideo`) para expor uma opção de habilitar/desabilitar tone-mapping por software.
