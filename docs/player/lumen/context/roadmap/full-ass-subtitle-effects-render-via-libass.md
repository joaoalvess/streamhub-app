**Dificuldade:** XL · **Relevância:** alta

## Abordagem proposta

Não existe `context/investigation/full-ass-subtitle-effects-render-via-libass.md` neste repositório (a pesquisa correspondente mais próxima está espalhada em três investigações já existentes: `word-by-word-subtitles.md` — karaokê `\k/\K/\kf` ignorado —, `use-fonts-embedded-in-the-video-to-render-subtitles.md` — sem registro de fontes anexadas — e `display-subtitles-with-hdr-effects.md` — legenda nunca compõe como bitmap sobre o vídeo). Esta pesquisa parte delas e de `docs/07-legendas.md` (mapa completo do subsistema atual).

**Status atual (confirmado no código).** O fork renderiza ASS com um parser Swift próprio (`AssParse`, `Sources/KSPlayer/Subtitle/KSParseProtocol.swift:39-138`) que lê `Dialogue:`/estilos do cabeçalho e converte um subconjunto de tags inline (`\an`, `\b`, `\i`, `\c`/`\1c`/`\3c`/`\4c`, `\shad`, `\fn`, `\fs`, `\u`, `\s`) em atributos de `NSAttributedString`, exibido como `Text`/`UILabel` (SwiftUI/UIKit) por cima do vídeo. Isso é **fundamentalmente diferente** de "renderizar ASS": não há suporte a `\move`, `\fad`/`\fade`, `\t` (transformações animadas), `\org`/`\pos` fora do numpad de alinhamento, `\clip`/`\iclip`, `\p` (desenho vetorial ASS), rotação 3D (`\frx/\fry/\frz`), karaokê, `ScaleX/ScaleY/Spacing` de estilo (comentados no código, nunca implementados) nem fontes embutidas do contêiner. Ou seja: hoje o player lê metadados ASS, mas não os **renderiza** no sentido usado por mpv/VLC/Jellyfin/Infuse — que todos usam a biblioteca **libass** como motor de composição, não um parser de texto caseiro.

**Achado central desta pesquisa: libass já está vendorizado e compilado no pipeline de build deste fork, mas nunca foi ligado ao Swift.** `kingslay/FFmpegKit` (a dependência SPM/CocoaPods deste projeto, `Package.swift:46`) já compila e expõe:
- Um produto SPM `libass` pronto (`FFmpegKit/Package.swift`: `.library(name: "libass", targets: ["libfreetype", "libfribidi", "libharfbuzz", "libass"])`), com os quatro `.xcframework` binários (`libfreetype`, `libfribidi`, `libharfbuzz`, `libass`) e um produto irmão `libmpv` (`FFmpegKit`+`libass`+`libmpv`, motor de playback alternativo baseado em mpv — fora de escopo aqui).
- No CocoaPods do Demo (`Demo/Podfile:7`, `Demo/Podfile.lock:35-37,66`), o pod `Libass` (0.17.1) já está resolvido e instalado ao lado de `FFmpegKit`/`OpenSSL`.
- Este próprio `Package.swift` (linhas 24-26) já tem, **comentado**, `.product(name: "Libass", package: "FFmpegKit")` — evidência de que alguém (provavelmente o autor no branch pago, ou um experimento abandonado) já cogitou ligar isso e recuou. **Atenção**: o nome do produto real no `FFmpegKit/Package.swift` é `libass` (minúsculo) — o comentário deste fork usa `Libass` (maiúsculo), que não resolveria como está; precisa correção ao reativar.
- Verificado via GitHub API que o repositório público `kingslay/KSPlayer` (mesmo upstream deste fork) tem exatamente os mesmos 4 arquivos em `Sources/KSPlayer/Subtitle/` e nenhuma referência a `libass`/`ASS_Renderer` em lugar nenhum — ou seja, **mesmo a hipótese de "a versão paga já faz isso" não tem evidência pública**; o vendoring binário parece existir só para alimentar o alvo `libmpv` (motor totalmente separado), não o `KSMEPlayer`/AVPlayer atual.

**A integração real segue o padrão usado por mpv (`sub/osd_libass.c`), VLC (`modules/codec/libass.c`) e o renderizador WebAssembly JASSUB — todos usam a mesma API C de libass, não há "outro jeito"**:

1. **Inicialização (uma vez por processo)**: `ass_library_init()` → `ASS_Library*`; `ass_renderer_init(library)` → `ASS_Renderer*`. Analogamente ao singleton implícito que hoje existe em `KSOptions.subtitleParses` (`KSParseProtocol.swift:20`), precisa de um wrapper Swift (`LibassRenderer`, novo arquivo) que possua esse ciclo de vida.
2. **Fontes**: `ass_set_fonts(renderer, defaultFontPath, defaultFamily, fontProviderMode, fontConfigPath, doFontConfigOverride)`. Sem inspecionar o binário `libass.xcframework` (não navegável pela API `contents` do GitHub — binário de release, não fonte), não dá para confirmar se foi compilado com fontprovider CoreText nativo ou puro fontconfig; `libfontconfig.xcframework` aparece como dependência do alvo `FFmpegKit` inteiro, não do alvo `libass` isolado, o que sugere que o produto `libass` sozinho pode não trazer fontconfig — exigindo fallback manual (fonte padrão embutida no app + fontes extraídas de anexos MKV via `ass_add_font(library, name, data, size)`, o mesmo mecanismo que a feature irmã "use fonts embedded in the video" precisa para extrair `AVMEDIA_TYPE_ATTACHMENT` de `MEPlayerItem.swift:283-317`).
3. **Carregar o script**:
   - **Arquivo `.ass`/`.ssa` externo**: `ass_read_memory(library, bytes, length, nil)` → `ASS_Track*`. Substituiria o parse via `AssParse`/`KSOptions.subtitleParses` (`KSParseProtocol.swift:20`) para esses dois formatos especificamente — SRT/VTT continuam no parser Swift atual, já que libass não é o formato nativo deles (embora aceite convertê-los via wrapper ASS mínimo, não vale o esforço).
   - **Faixa ASS embutida (MKV via FFmpeg)**: espelha o que `SubtitleDecode.init` (`Sources/KSPlayer/MEPlayer/SubtitleDecode.swift:22-33`) já faz ao alimentar `subtitle_header` no `assParse.canParse` — o equivalente libass é `ass_new_track(library)` + `ass_process_codec_private(track, subtitleHeaderBytes, size)`, seguido de `ass_process_chunk(track, rect.ass, length, startMs, durationMs)` por pacote decodificado, no mesmo ponto onde hoje `SubtitleDecode.text(subtitle:)` (`SubtitleDecode.swift:89-135`) despacha `rect.ass` para `assParse.parsePart`.
4. **Frame size**: `ass_set_frame_size(renderer, w, h)` (e `ass_set_storage_size` se a resolução de decodificação difere da de exibição) precisa ser chamado toda vez que o tamanho de exibição do vídeo mudar (rotação, resize de janela em macOS, troca de vídeo) — não existe hoje nenhum canal que exponha isso ao subsistema de legenda (`SubtitleModel` não conhece a geometria do vídeo).
5. **Render por tick**: `ass_render_frame(renderer, track, currentTimeMs, &detectChange)` devolve uma lista ligada `ASS_Image*` (bitmap alpha 8-bit por glyph run, com `bitmap`, `stride`, `w`/`h`, `dst_x`/`dst_y`, `color` RGBA pré-multiplicado). Isso é conceitualmente muito mais perto do caminho de **legenda bitmap já existente** (`SubtitlePart.image`/`origin`, usado hoje para VobSub/PGS/DVB via `VideoSwresample`+`CGImage.combine`, `Sources/KSPlayer/Core/Utility.swift:695-724`) do que do caminho de texto (`NSAttributedString`/`Text`/`UILabel`) usado para ASS hoje. A composição da lista de `ASS_Image` em uma única textura seria o novo trabalho.
6. **Frequência do tick — risco crítico de execução**: o callback que hoje alimenta `SubtitleModel.subtitle(currentTime:)` vem de um `Timer` de **0,1s / 10 Hz** (`Sources/KSPlayer/AVPlayer/KSPlayerLayer.swift:175`), pensado para atualizar barra de progresso — não para animar `\move`/`\fad`/`\t`/karaokê a 24-60fps. Sem trocar esse tick por algo casado ao `CADisplayLink` já existente para vídeo (`Sources/KSPlayer/MEPlayer/MetalPlayView.swift:79-83`), qualquer efeito animado do ASS ficaria visivelmente picotado a 10fps — a feature ficaria "certa mas trêmula". Isso é uma dependência de execução, não só um detalhe.

**Ponto de decisão de arquitetura**: rotear por **codec/extensão** (`.ass`/`.ssa` externo e faixas embutidas com codec ASS/SSA sempre via libass; SRT/VTT sempre pelo parser Swift atual) é a estratégia mais simples e é o que mpv/VLC/Jellyfin fazem — evita heurística de "detectar se o ASS usa efeitos avançados" para decidir motor.

- `Sources/KSPlayer/MEPlayer/SubtitleDecode.swift:16-33,89-135` — novo branch para `AV_CODEC_ID_ASS`/`AV_CODEC_ID_SSA` que roteia pacotes para `ass_process_chunk` em vez de (ou além de) `assParse`; `SubtitleDecode.init` (linhas 22-33) passa a alimentar `ass_process_codec_private` com o mesmo `subtitle_header` que hoje só vai para `assParse.canParse`.

- `Sources/KSPlayer/AVPlayer/KSPlayerLayer.swift:175` — o `Timer` de 10Hz precisa de uma via paralela de alta frequência (ligada ao `CADisplayLink` de `MetalPlayView.swift:79-83`) especificamente para quando o renderer libass estiver ativo.

- Dependência de extração de anexos: `Sources/KSPlayer/MEPlayer/MEPlayerItem.swift:283-317` (loop de streams do `AVFormatContext`) — precisa existir a extração de `AVMEDIA_TYPE_ATTACHMENT` (feature irmã ausente) para alimentar `ass_add_font`.

## Dependências (de outras features/upgrades)

- **"Use fonts embedded in the video to render subtitles"** (`context/investigation/use-fonts-embedded-in-the-video-to-render-subtitles.md`, Ausente) — pré-requisito de fidelidade: sem extração de anexos MKV e `ass_add_font`, libass cai em fontes de sistema/fallback para qualquer fonte customizada de fansub, que é o caso comum em anime.
- **Tick de exibição de alta frequência** (hoje 10Hz, `KSPlayerLayer.swift:175`) — pré-requisito para que efeitos animados (`\move`, `\fad`, `\t`, karaokê) não fiquem picotados; sem isso a feature entrega "tags corretas, movimento errado".
- **"Word-by-word subtitles" / karaokê** (`context/investigation/word-by-word-subtitles.md`, Ausente) — **é subsumida por esta feature**: libass já implementa nativamente `\k`/`\K`/`\kf`/`\ko`; adotar libass elimina a necessidade de reimplementar parsing de karaokê em Swift, reduzindo drasticamente o escopo daquela.
- **"Display subtitles with HDR effects"** (`context/investigation/display-subtitles-with-hdr-effects.md`, Ausente) — parcialmente subsumida: passar a compositar legenda como bitmap real sobre o vídeo (em vez de `UILabel`/`Text`) é pré-requisito comum às duas; uma vez com um compositor Metal de legenda, aplicar o mesmo `CAEDRMetadata`/colorspace do frame de vídeo é incremento natural.
- **Não depende do upgrade FFmpeg 6.1→8.x** (`context/investigation/ffmpeg-version-bundled.md`, Presente): libass é uma biblioteca C separada, versionada independentemente (0.17.1) dentro do mesmo pacote `FFmpegKit` — mas qualquer bump de versão do `FFmpegKit` deve preservar (ou também atualizar) o `libass.xcframework` vendorizado.
- **Compete por prioridade com as "joias da coroa"** (Dolby Vision nativo, HDR10+, Atmos, FFmpeg 8.x) — não bloqueia nem é bloqueada tecnicamente por elas, mas é um projeto de tamanho comparável (XL) disputando o mesmo caminho crítico de engenharia.

## Riscos e incógnitas

1. **Fonte real do produto SPM `libass` não inspecionável neste ambiente**: o repositório `kingslay/FFmpegKit` não está clonado localmente e a API `contents` do GitHub só devolve metadados de binaryTargets (`.xcframework`), não o header C real. Não foi possível confirmar sem baixar o pacote: (a) se `ass/ass.h` já vem com um module map Clang importável diretamente por `import libass` em Swift, ou se será necessário um target shim (`CLibass`) neste repositório; (b) se o binário foi compilado com fontprovider CoreText ou puro fontconfig — isso muda inteiramente a estratégia de fontes em tvOS (sandbox restrito).
2. **Nenhuma referência de código real dentro do ecossistema KSPlayer** (nem neste fork, nem no upstream público `kingslay/KSPlayer`, verificado via API do GitHub) mostra libass ligado ao `KSMEPlayer`/AVPlayer — só ao alvo `libmpv` (motor de playback alternativo e completamente diferente do usado aqui). A "referência pronta para copiar" não existe dentro do próprio projeto; vem inteiramente de fora (mpv/VLC/libass).
3. **Performance de composição em tempo real**: compor `ASS_Image` (bitmap CPU-side, um linked-list por frame) em cima de vídeo 4K HDR sem cair no antipadrão já documentado (`CGImage.combine`+PNG por frame, hoje só usado para subs de bitmap estáticos, nunca a 24-60fps) é o maior risco de execução — exige um caminho GPU dedicado que hoje não existe para nenhuma legenda no fork.
4. **tvOS sandbox e fontes**: o diretório de fontes de sistema não é exposto da mesma forma que em macOS/iOS; sem fontconfig customizado funcional, `ass_set_fonts`/`ass_fonts_update` pode falhar silenciosamente e depender só de fontes embutidas nos MKVs + uma fonte de fallback embutida no bundle do app.
5. **Threading e Sendable**: o projeto tem isolamento `MainActor` por padrão para tipos novos (nota de memória do usuário) e `SubtitleDecode.decodeFrame` roda hoje na thread de demux, não na main (`docs/07-legendas.md`, seção "Threading"). Chamar `ass_render_frame`/`ass_process_chunk` (API C, ponteiros opacos não-Sendable) a partir dessa thread — ou de uma thread de render Metal separada — exige um wrapper `nonisolated`/`@unchecked Sendable` cuidadoso ao redor de `ASS_Renderer`/`ASS_Track`, com a mesma classe de cuidado que `AssParse` já precisa hoje por ser stateful e compartilhado (`KSParseProtocol.swift:40-43`).
6. **Licenciamento**: libass é LGPL-2.1+, compatível com o fork GPL atual; o podspec do FFmpegKit usa `static_framework = true`, o que sob LGPL exige capacidade de relink (não bloqueia uso pessoal do StreamHub, mas relevante se o código for redistribuído).
7. **Escopo de "efeitos completos" é grande**: `\p` (desenho vetorial ASS, usado em karaokê complexo e efeitos de fansub), rotação 3D (`\frx/\fry/\frz`), `\clip` vetorial e `\t` com múltiplos parâmetros de easing são os itens de cauda longa mais difíceis de validar — mesmo com libass fazendo o trabalho pesado, a integração (fontes, timing, composição) para cobrir o catálogo real de fansubs de anime consumido pelo StreamHub é o que faz este item ser XL e não L.
