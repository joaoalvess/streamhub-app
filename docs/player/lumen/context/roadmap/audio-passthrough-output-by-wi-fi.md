**Dificuldade:** L · **Relevância:** baixa

## Abordagem proposta
Reaproveitar o pipeline de remux FFmpeg já presente no fork (`MEPlayerItem.startRecord(url:)`/`stopRecord()`, ver `context/investigation/record-video.md`, status Presente) para um segundo modo, restrito a uma única stream de áudio comprimido (AC3/EAC3/TrueHD/DTS), sem re-encode, com destino de rede (`udp://ip:port?...`) em vez de arquivo local. É exatamente o mecanismo que o próprio autor upstream do KSPlayer implementou como feature paga/custom "Remuxer Audio To URL" (kingslay/KSPlayer, issue #880): o app remuxa o pacote comprimido original — intocado — para um socket UDP; um receptor externo fora do app (Linux box, Mac/PC rodando mpv, ou hardware dedicado) recebe esse UDP, reconhece o formato e repassa via SPDIF/HDMI a um AV receiver, contornando a ausência de bitstream passthrough por HDMI no tvOS.

A saída local em PCM (`AudioRendererPlayer`/`AudioUnitPlayer`/`AudioGraphPlayer`/`AudioEnginePlayer`) permanece intacta e independente — o passthrough por Wi-Fi é um caminho paralelo de escrita de pacotes, não uma substituição do pipeline de decode/render existente. Como o pacote comprimido já está disponível no `AVFormatContext` de entrada antes de qualquer chamada a `avcodec_send_packet`, não é necessário mexer no decoder FFmpeg em si — só rotear o pacote da stream escolhida também para o `outputFormatCtx` de rede.

Dado que o tvOS 26 introduziu uma API nativa equivalente para passthrough via HDMI (`AVAudioContentSource.passthrough`, framework `AVFAudio`), mas seu status real para apps de terceiros segue incerto mesmo na tvOS 26.4 (fev/2026) — o que de fato foi entregue até agora foi "Continuous Audio Connection" (conexão contínua em Dolby MAT, ainda PCM, não bitstream) — a recomendação é tratar essa abordagem via Wi-Fi/UDP como tapa-buraco condicional: só vale a pena investir nela se, ao chegar a vez deste item no roadmap, a API nativa da Apple continuar não exposta/inutilizável a partir de uma arquitetura de decode custom via FFmpeg como a do KSPlayer.

## Arquivos e tipos afetados
- `Sources/KSPlayer/MEPlayer/MEPlayerItem.swift` — generalizar o mux já existente (`startRecord(url:)`/`stopRecord()`, hoje ~linhas 271-326 e 672-676, mapeamento de streams ~531-542) para um modo "single compressed audio stream, sem vídeo/legenda, sem re-encode", em paralelo ao modo de gravação em arquivo atual.
- `Sources/KSPlayer/AVPlayer/KSOptions.swift` — nova(s) flag(s) análoga(s) a `outputURL: URL?` (linha 42) para destino de rede (URL UDP + parâmetros de tuning), enable/disable por codec e delay manual.
- `Sources/KSPlayer/MEPlayer/KSMEPlayer.swift` — nova API pública paralela a `startRecord`/`stoptRecord` (~linhas 579-587) para iniciar/parar o passthrough de áudio independentemente da gravação em arquivo.
- Nenhuma mudança necessária em `FFmpegDecode.swift`/`AVFFmpegExtension.swift` (o pacote comprimido já existe antes do decode) nem nos 4 backends de saída de áudio (`AudioRendererPlayer.swift`, `AudioUnitPlayer.swift`, `AudioGraphPlayer.swift`, `AudioEnginePlayer.swift`), que continuam PCM-only para o alto-falante local.
- UI do StreamHub (fora deste pacote) — controle de delay de áudio (±5s) e de destino de rede, análogo ao que o app pago do KSPlayer expõe em "Ajustes avançados".

## Dependências
- Pipeline de remux/gravação já existente e usado em produção pelo demo do próprio fork (`context/investigation/record-video.md`, status Presente) — este roadmap é uma extensão dele, não uma feature isolada do zero.
- Um receptor de rede fora do controle do app (dispositivo com Linux/mpv/hardware dedicado capaz de transformar UDP em SPDIF/HDMI) — "bring your own receiver", exatamente como no app pago upstream.
- Tuning fino de parâmetros de UDP do FFmpeg (`bitrate`, `burst_bits`, `pkt_size`) por sessão/rede, sensível a variação de banda.

## Riscos e incógnitas
- Sincronia A/V: o áudio remuxado chega ao receptor adiantado em relação ao vídeo renderizado localmente, pois o pipeline de PCM local ainda buffereia/pacinga a reprodução — exige controle manual de delay (±5s); não há solução automática conhecida nem no app pago (confirmado na própria issue upstream #880).
- Confiabilidade do UDP: no relato de desenvolvimento do próprio KSPlayer, TrueHD/DDP Atmos de alto bitrate sofrem perda de pacote e cortes a menos que `bitrate`/`burst_bits`/`pkt_size` sejam ajustados manualmente por rede — não há um valor default que funcione universalmente.
- Troca de faixa de áudio em runtime não é limpa: o player receptor (mpv, no relato upstream) precisa ser reiniciado manualmente após troca de audio track, pois o formato do bitstream muda sem renegociação — ficaria a cargo de cada receptor externo lidar com isso, fora do controle do StreamHub.
- Depende de hardware externo dedicado ao usuário — contraria a ambição de uma experiência 100% nativa/polida só no Apple TV, que é o objetivo final do projeto (paridade com Infuse/versão paga via MKV+Dolby Vision nativo, HDR10+, Atmos nativo, tudo dentro do próprio tvOS).
- Status real da API nativa da Apple (`AVAudioContentSource.passthrough`, framework `AVFAudio`) é incerto e as fontes disponíveis se contradizem: cobertura de imprensa (AppleInsider, FlatpanelsHD, cecritic) trata a chegada do passthrough via HDMI como certa a partir da tvOS 26, mas comunidades de desenvolvedores que acompanham de perto (thread da Firecore/Infuse, issues do Swiftfin #1563/#1641) não confirmam adoção real por apps de terceiros até tvOS 26.4 — o que de fato lançou nessa versão foi "Continuous Audio Connection" (Dolby MAT contínuo, ainda PCM), uma feature relacionada mas distinta de bitstream passthrough puro. Não foi possível confirmar, com uma fonte posterior a fev/2026, se isso mudou.
- Mesmo que a Apple libere a API nativa de fato, não está claro se ela é acessível a partir de uma arquitetura de decode custom via FFmpeg que já decodifica tudo para PCM antes de qualquer saída (como o KSMEPlayer) — pode exigir repassar o pacote comprimido original a um novo `AVAudioContentSource`, uma mudança arquitetural de porte comparável (ou maior) à do próprio hack de rede aqui descrito. Se a API nativa se confirmar viável, ela resolveria o problema de origem (falta de passthrough HDMI) de forma mais nativa e tornaria essa abordagem via Wi-Fi obsoleta antes mesmo de ser lançada no StreamHub.
- Risco de política de App Store: o próprio autor original do KSPlayer especulou sobre a Apple poder proibir esse tipo de tunelamento de áudio bruto por rede; não é uma feature de app irmão nem amplamente testada — o próprio maintainer upstream a classificou como "muito nicho" e só a implementou mediante pedido/pagamento avulso de um usuário.
- Não relevante para os objetivos-âncora do projeto (MKV+Dolby Vision nativo via HLS local→AVPlayer, HDR10+, Atmos nativo, FFmpeg 8.x) — é uma solução de contorno para um problema de plataforma (falta de passthrough HDMI nativo) que a própria Apple está endereçando por outra via, ainda que com timeline incerta.
