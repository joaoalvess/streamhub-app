**Dificuldade:** L (o MVP restrito ao motor AVPlayer/HLS local pode ser M; a versão completa cobrindo também o motor FFmpeg/MEPlayer é L) · **Relevância:** alta

## Abordagem proposta

Não existe investigação dedicada para este item (README lista "Video switching with zero delay" com "—", sem `context/investigation/*.md` próprio) e não há nenhuma infraestrutura de prewarm/dual-pipeline no fork hoje — confirmado por busca ampla (`second player`, `dual`, `parallel`, `prewarm`, `prebuffer`, `hot-swap`) sem nenhum resultado relevante em `Sources/KSPlayer`.

**O que "switching" significa neste projeto.** Não é troca de trilha de áudio/legenda dentro do MESMO arquivo já aberto — isso já é quase instantâneo hoje via `MEPlayerItem.select(track:)` (`Sources/KSPlayer/MEPlayer/MEPlayerItem.swift:134-158`, doc 03 "Pegadinhas"): só alterna `stream.discard` e reseeka localmente, sem reabrir rede. O alvo real, conforme a diretriz do dono ("troca de stream sem delay"), é trocar para uma **URL/arquivo diferente** que representa o mesmo título — outro candidato de stream de debrid (qualidade/fonte diferente), um fallback automático quando o stream atual trava, ou a troca dublado↔legendado (perfis `casual`/`cinema` do StreamHub, que resolvem para arquivos distintos, ver `StreamHub/Streams/StreamProfile.swift` e `StreamHub/Playback/PlaybackCoordinator.swift`) — sem o usuário ver tela preta/spinner por vários segundos.

**Diagnóstico da lentidão atual (confirmado no código, não só suposto).** Tanto o motor FFmpeg quanto o nativo fazem *shutdown antes* de abrir a nova URL — teardown e reopen síncronos, sequenciais, sem qualquer sobreposição:
- `KSPlayerLayer.url` didSet (`Sources/KSPlayer/AVPlayer/KSPlayerLayer.swift:128-159`): mesmo engine + URL nova → `stop()` (que chama `player.shutdown()`, `:318-330`) → `player.replace(url:options:)` → `prepareToPlay()` do zero.
- `KSMEPlayer.replace(url:options:)` (`Sources/KSPlayer/MEPlayer/KSMEPlayer.swift:317-334`): primeira linha do corpo é `shutdown()`, só depois `playerItem = MEPlayerItem(url:options:)` — o novo demuxer FFmpeg (`avformat_open_input`+`avformat_find_stream_info`, doc 03 "Construção e abertura") só começa a existir depois que o antigo já foi destruído.
- `KSAVPlayer.replace(url:options:)` (`Sources/KSPlayer/AVPlayer/KSAVPlayer.swift:448-453`): mesmo padrão — `shutdown()` primeiro, `AVURLAsset` novo depois.
- Precedente de UX já existente no fork (mas não usado pelo StreamHub, que é SwiftUI/tvOS): `VideoPlayerView.change(definitionIndex:)` (`Sources/KSPlayer/Video/VideoPlayerView.swift:355-367`) — um seletor de "qualidade" que guarda `currentPlaybackTime`, chama `set(url:)` (o mesmo caminho cold acima) e só depois faz `seek(time:)` para o tempo salvo. Mostra que o *padrão de UX* (trocar de fonte preservando posição) já foi pensado no fork em algum momento — só nunca com um mecanismo que evite o gap.
- O próprio fork já instrumenta onde o tempo de abertura vai: `KSOptions.dnsStartTime/tcpStartTime/tcpConnectedTime/openTime/findTime/readyTime` (`KSOptions.swift:94-105`), consumidos no dict de log de `KSPlayerLayer.changeLoadState` (`:404-420`). Isso é o ponto de partida natural para medir, com hosts de debrid reais do StreamHub, quanto cada etapa (DNS, TCP/TLS, HTTP GET, probe FFmpeg, primeiro frame decodificado) realmente custa antes de desenhar a solução completa.

**Proposta em camadas incrementais** (da mais barata à mais completa — não precisam ser todas implementadas de uma vez):

1. **Prewarm de rede (S–M).** Antes do usuário confirmar a troca (ex.: enquanto navega um seletor de fontes/qualidades, ou assim que o addon retorna a lista de candidatos), abrir uma conexão HTTP antecipada para a(s) URL(s) candidata(s) — `URLSession` com Range/HEAD request, ou uma extensão de `AbstractAVIOContext` (`Sources/KSPlayer/AVPlayer/PlayerDefines.swift:347-378`) — só para resolver DNS + handshake TCP/TLS com antecedência. Reduz `dnsTime`+`tcpTime` do `openTime` total; não elimina probe/demux/decoder init. Contido, baixo risco, não mexe em threading do core.
2. **Pre-roll via `AVQueuePlayer` (M, cobre o motor AVPlayer).** `KSAVPlayer` já envolve um `AVQueuePlayer` (`KSAVPlayer.swift:226`) — a mesma classe que o próprio fork usa para loop gapless via `AVPlayerLooper` (`:300-324`). A Apple documenta `insert(_:after:)` exatamente para permitir que o próximo item comece a carregar/pré-rolar enquanto o atual toca. Ao invés de só usar `replaceCurrentItem` (`:300-324`), criar o `AVPlayerItem` do candidato assim que ele for conhecido, inserir com `insert(_:after: currentItem)`, observar KVO `status`/`isPlaybackLikelyToKeepUp` até `.readyToPlay`, e ao confirmar a troca usar `advanceToNextItem()` (ou reordenar a fila) em vez de recriar tudo. Esse é o caminho relevante para a "joia da coroa" do roadmap — MKV com Dolby Vision via remux HLS local → AVPlayer — porque ambos os candidatos, sendo HLS locais, são leves de pré-carregar.
3. **Pre-open paralelo do demuxer FFmpeg (M–L, cobre o motor MEPlayer).** Inverter a ordem em `KSMEPlayer.replace`: nunca fazer `shutdown()` antes. Instanciar um segundo `MEPlayerItem` para a URL candidata em paralelo — a classe não depende de estado global além do `onceInitial` estático (`MEPlayerItem.swift:94-121`), então duas instâncias são tecnicamente instanciáveis hoje — e deixá-lo abrir/ler até acumular buffer suficiente (reaproveitando `KSOptions.playable(capacitys:isFirst:isSeek:)`), tudo enquanto o item atual segue tocando na tela. Só ao ficar pronto (ou o usuário confirmar), fazer o hot-swap: fechar o item antigo, religar `videoOutput.renderSource`/`audioOutput.renderSource` para o novo `playerItem` (o padrão de religar já existe parcialmente em `replace(url:)`, linhas 330-333) e resincronizar os clocks (`KSClock`, `KSOptions.swift:670-682`) para o tempo alvo.
4. **Continuidade de saída (comum às duas engines).** Hoje o hand-off de vídeo recria a view e invalida a antiga no `didSet` de `videoOutput` (`KSMEPlayer.swift:22-28`), e o hand-off de áudio (`audioOutput.prepare(audioFormat:)`) só roda uma vez, no open (doc 05; doc 03 "Ordem de inicialização do áudio"). Trocar para um candidato com formato de áudio diferente (ex.: stereo AAC → 5.1 EAC3) em pleno playback não tem caminho pronto — precisa de reconfiguração em runtime sem estalo/silêncio perceptível.

**Definição de sucesso**: não é zerar a latência de rede contra hosts remotos de debrid (fisicamente impossível), e sim eliminar o gap perceptível na UI (tela preta, spinner, "Carregando…") — só trocar quando o novo pipeline já tem primeiro frame decodificado e áudio pronto, o mesmo princípio por trás de troca de qualidade ABR em players de streaming maduros (o player nunca solta o frame atual até o novo estar pronto).

## Arquivos e tipos afetados

- `Sources/KSPlayer/AVPlayer/KSPlayerLayer.swift` — `url` didSet (128-159, hoje sempre `stop()`+reopen), `set(url:options:)`/`set(urls:)` (261-277), `stop()` (318-330), `prepareToPlay()` (488-493): precisa de um novo caminho "switch sem stop" que mantenha o player atual tocando durante a preparação do candidato.
- `Sources/KSPlayer/MEPlayer/KSMEPlayer.swift` — `replace(url:options:)` (317-334, shutdown síncrono antes do novo item), `videoOutput` didSet (22-28, invalida a view antiga), `playerItem` (18, hoje uma única instância, sem noção de "próximo item").
- `Sources/KSPlayer/MEPlayer/MEPlayerItem.swift` — praticamente todo o ciclo de open/read (`openThread` 164+, `createCodec` 328+, `MESourceState` em `Model.swift:17`): para permitir duas instâncias vivas simultâneas com segurança, é preciso revisitar o retain cycle intencional no close (`:633-634`, hoje pensado para uma instância fechando por vez) e os pontos de estado global/estático descritos no doc 03 ("Pegadinhas" — threading, `Sendable` de fachada).
- `Sources/KSPlayer/AVPlayer/KSAVPlayer.swift` — `replace(url:options:)` (448-453, mesmo padrão shutdown-then-new), `replaceCurrentItem(playerItem:)` (300-324, já convive com `AVPlayerLooper`/`AVQueuePlayer` — base pronta para `insert(_:after:)`), `player: AVQueuePlayer` (226).
- `Sources/KSPlayer/AVPlayer/KSOptions.swift` — métricas de abertura já existentes (`dnsStartTime`/`tcpStartTime`/`tcpConnectedTime`/`openTime`/`findTime`/`readyTime`, 94-105): reaproveitar para instrumentar e decidir quando um candidato está "pronto o bastante" para o swap; novos hooks/flags de prewarm.
- `Sources/KSPlayer/MEPlayer/AudioEnginePlayer.swift` (e os demais 3 backends de áudio, doc 05) — `prepare(audioFormat:)` só é chamado no open; precisa suportar reconfiguração em runtime quando o candidato tem formato de áudio diferente do atual.
- `Sources/KSPlayer/Video/VideoPlayerView.swift` — `change(definitionIndex:)` (355-367): precedente de UX já existente no fork (UIKit, não usado pelo StreamHub), útil como referência de fluxo (seletor de qualidade preservando `currentPlaybackTime`), mas usa o caminho cold; não é mecanismo a copiar, só UX a igualar em resultado.
- Camada de app StreamHub (fora deste pacote, `/Users/joaoalves/Developer/streamhub`) — `StreamHub/Playback/PlaybackCoordinator.swift`: hoje só monta a URL escolhida e delega ao Infuse externo via URL scheme (`playViaInfuse`/`playEpisodeViaInfuse`, ambos chamando `InfuseLauncher.open(url)`); esta feature só tem onde existir quando o player nativo deste fork substituir o Infuse como motor de playback do StreamHub. É o app quem vai expor a UI de "trocar fonte"/candidatos e decidir *quando* pré-aquecer o próximo (ex.: ao listar `sources(for:mode:)`, já pré-aquecer o primeiro candidato preferido).

## Dependências

- **Depende (forte) de o player nativo deste fork virar o motor de playback real do StreamHub.** Hoje `PlaybackCoordinator.playViaInfuse`/`playEpisodeViaInfuse` delegam 100% ao Infuse externo via deep link (`StreamHub/Playback/PlaybackCoordinator.swift:186-240,242-316`) — não há player deste fork em produção no app ainda. Sem isso, não existe onde plugar a troca sem delay.
- **Se beneficia (mas não bloqueia) do precache/buffering em disco** (`context/investigation/precache-data-to-hard-drive.md`, status ausente) — um cache local dos primeiros segundos dos candidatos tornaria o pre-open ainda mais barato/rápido.
- **Compartilha infraestrutura com as outras duas prioridades de usabilidade do dono** (seek com preview, buffering/precache): as três se beneficiam de uma capacidade comum de "um segundo pipeline de demux decodificando em paralelo, sem afetar o playback principal" — vale desenhar essa capacidade uma vez e reaproveitar (seek preview também precisa decodificar frames de um ponto arbitrário sem interromper o playback atual).
- **Ortogonal ao upgrade de FFmpeg 6.1→8.x e ao trabalho de Dolby Vision/HDR10+/Atmos nativos** — não depende deles, mas qualquer refator de threading em `MEPlayerItem` feito aqui precisa ser coordenado com quem estiver mexendo em decode/HDR ao mesmo tempo, já que ambos tocam os mesmos arquivos centrais (`MEPlayerItem.swift`, `MEPlayerItemTrack.swift`).

## Riscos e incógnitas

- **Maior risco arquitetural**: o design atual de `MEPlayerItem`/`KSMEPlayer` assume posse exclusiva de um `AudioOutput`/`VideoOutput`/par de clocks por instância (doc 03), e o shutdown tem um retain cycle intencional (`MEPlayerItem.swift:633-634`) desenhado para uma instância fechando por vez. Rodar duas instâncias vivas simultaneamente (uma tocando, uma pré-abrindo) exige auditar cada ponto de estado global/estático (`onceInitial`, o timer de 50 ms no main run loop, nomes fixos de thread) para garantir que não colidem entre si.
- **Handoff de áudio sem estalo**: trocar o pipeline de saída para um formato diferente (ex.: stereo AAC → 5.1 EAC3) em pleno playback não tem precedente no fork — hoje só troca de rota com o *mesmo* formato é tratada (doc 05, `KSMEPlayer.swift:168-189`).
- **Sincronização de tempo**: a troca precisa preservar o `currentPlaybackTime` exato; como os dois pipelines têm clocks (`KSClock`) independentes, o seek do candidato precisa acontecer *durante* o pre-open (antes do swap) — diferente do padrão atual em `change(definitionIndex:)`, que seeka *depois* do reload.
- **Custo de rede/CPU duplicado**: manter dois demuxers FFmpeg abertos ao mesmo tempo (mesmo throttled) consome banda e CPU extras exatamente quando a rede já é o gargalo (streams HTTP de debrid, muitas vezes fora da rede local) — precisa de uma política clara de quando vale a pena pré-abrir (ex.: só quando o usuário abre o seletor de fontes, não especulativamente o tempo todo).
- **Limite de sessões de hardware decode no tvOS**: dois pipelines de vídeo ativos ao mesmo tempo (Metal/`AVSampleBufferDisplayLayer` + VideoToolbox) podem esgotar sessões de decode simultâneas no Apple TV — comportamento plausível (VideoToolbox historicamente limita sessões concorrentes por processo/dispositivo), mas **não confirmado nesta pesquisa** quantas sessões o hardware atual do Apple TV suporta; validar na prática antes de assumir que dois streams decodificando ao mesmo tempo é viável no dispositivo alvo.
- **Nenhuma fonte pública descreve como o Infuse (closed-source) ou a versão paga do KSPlayer implementam essa troca especificamente.** Busca nos issues públicos do `kingslay/KSPlayer` não retornou nenhuma discussão dedicada a "switch de stream sem delay"; não há uma receita pronta para copiar além do padrão genérico `AVQueuePlayer.insert(_:after:)` documentado pela própria Apple para playlists/gapless.
- **MVP mais barato**: a camada 2 (pre-roll via `AVQueuePlayer`) sozinha provavelmente entrega a maior parte da percepção de "zero delay" para o caminho HLS-remux→AVPlayer — que é justamente a joia da coroa do roadmap (MKV+DV nativo via remux HLS local). O caminho MEPlayer/FFmpeg puro (hoje usado para tudo que HLS/AVPlayer não decodifica nativamente) é o que exige o trabalho arquitetural mais pesado (camada 3) — priorizar nessa ordem reduz risco.
